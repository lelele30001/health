

第三章	系统需求分析与详细设计
3.1	可行性分析
3.1.1	技术可行性
本系统采用前后端分离的架构设计，前端使用 Vue 3 框架结合 Element Plus 组件库，后端采用 SpringBoot 框架结合 MyBatis-Plus 操作 MySQL 数据库。这套技术栈成熟稳定，有大量的文档和社区支持，能够满足系统的开发需求。前端技术方面，Vue 3 提供了响应式数据绑定和组件化开发能力，Element Plus 提供了丰富的 UI 组件，可以快速构建用户界面。ECharts 用于数据可视化，WangEditor 用于富文本编辑。后端技术方面，SpringBoot 简化了 Spring 应用的开发，提供了自动配置和快速启动能力。MyBatis-Plus 提供了强大的 ORM 功能，简化了数据库操作。Apache Shiro 实现了安全认证和权限管理。这些技术组合能够满足系统的功能需求。
3.1.2	市场可行性
当前大众健康意识不断提升，健身人群对科学膳食管理、食谱分享与个性化饮食推荐的需求持续增长，而市面上多数膳食类平台功能单一，缺乏针对健身目标的精准推荐服务，市场存在明显需求缺口。本系统面向健身人群打造，聚焦膳食分享、营养查询、个性化食谱推荐等核心需求，操作简单、实用性强，能够满足学生、健身爱好者等群体的日常饮食管理需要。同时，系统基于开源技术开发，整体成本较低，无需高额部署费用，适合面向普通用户免费使用，具备明显的性价比优势。同时系统采用轻量化设计，使用门槛低，可在校园、社区、健身工作室等多种场景推广应用。与功能繁杂的商业健康软件相比，本系统主题明确、界面简洁、专注核心服务，更符合普通健身用户的使用习惯。结合市场需求、目标用户定位与轻量化优势，本系统具备良好的市场落地可行性。
3.2	功能性需求分析
3.2.1	用户功能需求分析
本系统主要面向两类用户：普通用户和管理员。普通用户可以通过系统进行膳食分享、查看推荐、参与社区互动等操作；管理员负责系统内容管理、用户管理和数据统计等工作。
(1)	普通用户功能需求
用户可通过登录注册模块完成账号注册、登录，注册时填写基本信息并完成校验，登录后由 Token 保障身份安全；进入个人中心可查看、编辑个人资料，管理健康档案、健身目标，还能查看自己发布的内容与收藏记录，为个性化推荐提供依据。食谱浏览模块支持食谱列表查看、搜索及详情查看，用户可收藏、点赞、踩和评论；食谱推荐模块结合用户健身目标与历史行为，智能推送符合营养需求的个性化食谱。社区互动模块方便用户发布、浏览、点赞和评论膳食相关帖子，分享健康经验；AI 助手模块则可实时为用户解答膳食营养、健康饮食等问题，整体功能完整、操作简单，能满足健身人群日常膳食管理与交流需求。
(2)	管理员功能需求
管理员后台具备完善的管理能力，可对用户、食谱、食材、帖子及评论进行统一管控。在用户管理中，管理员能够查看与搜索用户信息，并查阅用户健康档案及发布内容，保障信息真实合规；食谱与食材支持列表查看、搜索，方便快速核对信息；内容与评论管理可对社区帖子和评论进行查询、清理，及时处理违规内容，维护平台交流秩序。同时，系统提供数据统计功能，以可视化形式展示用户总量、性别、年龄、健身目标等分布数据，便于管理员直观掌握平台运行状况，为后续优化提供依据。
3.3	非功能性需求
3.3.1	性能需求
系统采用 SpringBoot 内嵌 Tomcat 服务器，支持多线程处理并发请求。前端页面采用 Vue 3 的虚拟 DOM 和异步渲染机制，提升页面响应速度。数据库查询使用 MyBatis-Plus 的延迟加载和分页查询机制，减少单次查询的数据量，提高查询效率。系统通过 RESTful API 与前端通信，采用 JSON 格式传输数据，数据量小、解析速度快。
3.3.2	安全性需求
系统采用 Token 机制进行用户身份认证，登录成功后生成 Token 并在后续请求中携带，确保用户身份安全。系统使用 MD5 对用户密码进行加密存储，防止密码泄露。MyBatis-Plus 的参数绑定机制能够有效防止 SQL 注入攻击。前端对用户输入进行基本校验，后端同样进行输入验证，双重防护提高系统安全性。
3.3.3	易用性需求
系统基于 B/S 架构设计，用户通过浏览器即可访问，无需安装额外软件。界面采用 Element Plus 组件库设计，风格统一、美观大方。系统在各功能模块提供清晰的导航和操作提示，帮助用户快速上手。表单输入提供实时校验和错误提示，提升用户体验。
3.4	详细设计
3.4.1	系统总体结构设计
本系统采用前后端分离的 B/S 架构，前端和后端通过 RESTful API 进行数据交互。前端负责页面展示和用户交互，后端负责业务逻辑处理和数据存储。
(1)	系统架构设计
系统主要由用户认证、个人中心、食谱浏览、社区互动、AI 助手和管理后台模块组成。当用户登录时，通过传输层 HTTP/HTTPS 请求和响应，使用 Apache Shiro 对权限安全进行验证，严格控制权限访问及系统安全。应用层为不同功能提供相应的应用服务，数据层使用 MySQL 数据库存放业务数据，提高系统的响应速度。系统架构图如图3-1所示：。

3.5.2	推荐算法设计
系统采用混合推荐算法，结合了协同过滤和基于内容的推荐策略，参考了 FitRec（Fitness Recommendation）领域的研究框架，将用户身体数据和健身目标作为核心特征，用于实现个性化饮食推荐与身体数据趋势分析。根据用户的健身目标、健康档案和历史行为，为用户推荐个性化的食谱。
1.	用户特征提取：
(1)	从用户档案表中获取健身目标（增肌、减脂、维持）、饮食禁忌等信息
(2)	从用户行为表中提取偏好信息，包括点赞、收藏、浏览等行为
(3)	计算用户对不同食谱的偏好权重（收藏：10分，点赞：5分，浏览：1分）
2.	食谱特征提取：
(1)	从食谱表中获取健身目标、营养标签（高蛋白、低卡、均衡等）、饮食禁忌等信息
(2)	计算食谱的热度分数（基于点赞、收藏、浏览的加权总和）
(3)	提取食谱的发布时间，用于推荐时优先考虑较新的内容
3.	相似度计算：
(1)	协同过滤相似度：计算目标用户与其他用户的行为相似度（使用余弦相似度），根据相似用户的行为推荐食谱
(2)	健身目标匹配：计算用户健身目标与食谱健身目标的匹配度（完全匹配：100分，维持用户对所有目标：90分，其他目标对维持：70分，不匹配：0分）
(3)	营养标签匹配：根据用户健身目标计算与食谱营养标签的匹配度（如增肌用户对高蛋白标签：100分，减脂用户对低卡标签：100分）

4.	推荐生成：
(1)	对于有行为的用户：协同相似度 40% + 健身目标匹配 30% + 营养标签匹配 20% + 热度 10%
(2)	对于冷启动用户：健身目标匹配 60% + 营养标签匹配 40%
(3)	过滤用户自己发布的帖子和已经有行为的帖子
(4)	过滤不符合用户饮食禁忌的帖子
(5)	按推荐分数排序，生成Top 15推荐列表

5.	推荐存储与更新：
(1)	将推荐结果存储到recipe_recommendation表中
(2)	当用户行为发生变化时，重新生成推荐结果

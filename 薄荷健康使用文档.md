## 通用规范

### 域名

- `https://fc.boohee.com` （除图片识别接口外，都使用该域名）

### 访问令牌

除获取访问令牌接口外，所有接口访问需要带访问令牌 AccessToken 访问令牌可以通过下面三种形式之一传递

- Http Header，字段名 AccessToken
- Query Parameters 或 Post Data，参数名 AccessToken
- Http Cookie，cookie 名 AccessToken

访问令牌通过 `获取访问令牌` 接口获得

### 结果输出

默认返回标准 json 格式数据

### 异常处理

异常分业务失败和错误

- 业务失败 http code 返回 200，通常格式为 {success: 0, message: 'xxxx'}
- 错误返回 http code 4xx 或 5xx，通常格式为 {error: {code: 1001, message: 'xxxx'}}

#### 错误 code 列表

| code | message                  |
| :--- | :----------------------- |
| 1001 | 缺少 app_id 或 app_key   |
| 1002 | 无效的 app_id 或 app_key |
| 1003 | 非法sign值               |
| 1010 | 缺少 AccessToken         |
| 1011 | AccessToken 已过期       |
| 1012 | 无效的 AccessToken       |
| 1013 | 缺少参数或参数不规范     |
| 1014 | 非法请求时间戳           |
| 1021 | 找不到该食物             |
| 1024 | 未获授权的接口           |

# Sign算法说明

举例：假设请求参数键值对如下

```yml
app_id: boohee
timestamp: 1470042310
```

## 第一步

对数所有请求参数按照键名进行正序排序，排序结果如下，注意要保持键值对的对应关系且app_key不参与运算。

```yml
app_id: boohee
timestamp: 1470042310
```

## 第二步

循环对每个键值进行拼接处理，拼接成如下的字符串：`app_idbooheetimestamp1470042310`

## 第三步

计算sign

1.获取 app_key， 假如当前实例中的 app_key 为 12345

2.将app_key拼接在第二步的结果的前后， 注意没有任何拼接符，结果如下： `12345app_idbooheetimestamp147004231012345`

3.将上述字符串进行md5加密后即可得到sign，要求MD5方法返回的是32位小写的MD5值，计算结果如下： 27f61732856d133758e7f8927a3abed1

4.将计算的sign作为一个请求参数放到参数列表中

```yml
app_id: boohee
timestamp: 1470042310
sign: 27f61732856d133758e7f8927a3abed1
```



## 获取访问令牌（新版）

获取访问令牌接口，相比较 v1 版的接口，增加了接口 sign 值校验，以及避免了 app_key 的明文传输。

#### 接口信息

| 名称     | 说明                  |
| :------- | :-------------------- |
| 接⼝地址 | /api/v2/access_tokens |
| 接⼝说明 | 获取接口 access_token |
| 请求方法 | POST                  |
| 响应格式 | json                  |

#### 请求参数

| 参数名      | 类型    | 必填 | 描述                                                         | 示例值                           |
| :---------- | :------ | :--- | :----------------------------------------------------------- | :------------------------------- |
| app_id      | string  | 是   | 注册平台后获取                                               | boohee                           |
| app_user_id | string  | 否   | 厂商自定义用户ID（确保唯一）                                 | uuid-xxxxx-xxxx-xxxx-xxx         |
| timestamp   | integer | 是   | Unix时间戳(北京时间)                                         | 1635231332                       |
| sign        | string  | 是   | sign 值，生成方式参考：[Sign算法说明](https://fc.boohee.com/panel/open_api_docs/dc7d9153) | 27f61732856d133758e7f8927a3abed1 |

#### 请求返回

```json
{
  "access_token": "xxx",  
  "expired_at": "2017-05-08 15:27:33 +0800"
}
```

备注：

- 访问 `获取访问令牌` sign 值的生成需要 app_key 参数，请求方务必保证 app_key 安全存放，建议只放在服务器端保存（通过存放在系统环境变量上，并保持 配置信息在代码库中脱敏），只在服务器端调用 `获取访问令牌` 接口，客户端通过委托服务端调用获取 access_token。
- expired_at 表示令牌过期时间，默认过期时间在 1 个月后，请在过期前重新获取新的 access_token。

## 食物列表

#### 接口信息

| 名称     | 说明                 |
| :------- | :------------------- |
| 接⼝地址 | /api/v1/foods        |
| 接⼝说明 | 根据分类查询食物列表 |
| 请求方法 | GET                  |
| 响应格式 | json                 |

#### 请求参数

| 参数名    | 类型   | 必填 | 描述                                                         | 示例值 |
| :-------- | :----- | :--- | :----------------------------------------------------------- | :----- |
| kind      | string | 是   | 分类种类，取值为 group / brand / cooking / season 之一       | brand  |
| value     | string | 是   | 分类 id                                                      | 1      |
| sub_value | string | 否   | 二级分类 id                                                  | 2      |
| page      | string | 否   | 返回指定页的结果，默认取第 1 页                              | 1      |
| order_by  | string | 否   | 1 表示常见， 2 表示热量，默认是 1                            | 2      |
| food_type | string | 否   | 选择要返回的食物类型。 基础食材: 0， 包装食物: 1， 菜肴: 2， 可组合数字返回所需类型。 如：food_type 为 0 返回基础食材，food_type 为 02 返回基础食材和菜肴 | 02     |

```json

  total_pages: 0,
  foods: []

```

备注：

- page 表示当前，total_pages 是总页数，默认每页返回 30 条数据
- 最多显示前10页，page 最大值为10
- kind、value、sub_value 取值参照接口 `4.食物分类.`
  - group 食物类别
  - brand 食物品牌
  - cooking 食物做法
  - season 食物季节
- food 中相关字段
  - id 食物 id
  - code 食物的 code，通过 code 查询食物详情
  - name 食物名称
  - thumb_image_url 图片完整 url，可直接显示并缓存
  - is_liquid 是否为液体
  - health_light 定义 0 没灯 1 绿灯 2 黄灯 3 红灯
  - weight 食物重量(g)
  - calory 卡路里(千卡)
  - 其他营养元素请查看营养元素对照表
